Top 10 Pitfalls + Fixes

  1. Premature microservices. Smell: lock‑step deploys; chatty calls. Fix: consolidate into one deployable per cohesive capability; keep API boundary; re‑split later.
  2. Shared databases across services. Smell: cross‑service writes; implicit invariants. Fix: single‑writer ownership; read models/CDC for others; contract tests.
  3. Weak SLIs. Smell: CPU “green” while users complain. Fix: measure client‑visible latency/success; add freshness where relevant; tie alerts to burn rate.
  4. Cache stampedes. Smell: origin spikes on TTL boundary. Fix: jitter TTLs, request coalescing, background refresh, and cache tags.
  5. Hot partitions. Smell: one shard’s p99 far worse. Fix: better shard keys, salting, adaptive rebalancing; backfills with idempotency.
  6. Missing idempotency. Smell: duplicate orders on retries. Fix: idempotency keys on write endpoints; de‑dup at sinks.
  7. Unsafe rollouts. Smell: “all‑or‑nothing” deploys. Fix: canary steps with SLI gates; 1‑click rollback; expand/contract migrations.
  8. Telemetry bloat. Smell: soaring log bills; slow queries. Fix: structured logs, sampling, metric cardinality caps, tiered retention.
  9. Secrets leakage. Smell: keys in repo/CI logs. Fix: secret manager or workload identity; rotation and audit; pre‑commit scanners.
  10. Unrehearsed incidents. Smell: chaos during chaos. Fix: roles (IC/Comms/Op/Scribe), templates, and game days with abort criteria.

Appendix B — Progression Map (Beginner → Intermediate → Advanced)

The progression synthesizes the study plan into a concrete ladder. Use it to plan personal growth or to structure a team upskilling program.

B.1 Beginner → Intermediate (this book)

  • Outcome: name architecture pieces; draw simple diagrams; explain baseline trade‑offs.
  • Bridge tasks: implement a microservice + DB; add tracing; run a small load test; write your first SLOs.
  • Milestone for entry: you can design service boundaries, select storage/caching, author SLOs, deploy with CI/CD, and run one fault‑injection test.

B.2 Intermediate → Advanced

  • Outcome: design multi‑region, high‑scale systems; perform capacity planning; run platform components.
  • Bridge tasks:
    • Multi‑region replication exercise (active‑passive with failover drills).
    • Capacity planning with unit economics and a telemetry budget.
    • Platform engineering mini‑project (self‑service templates, CI/CD, SLO dashboards).
  • Milestone for entry: you can demonstrate a production‑like capstone (Ch9) and articulate trade‑offs with data.

B.3 Review & learning cadence

Adopt a cadence that matches real‑world teams:

  • Weekly: one‑page update + a passing test + tiny demo.
  • After Ch3, Ch5, Ch8: design review sessions (data, reliability, patterns).
  • Per milestone: risk burndown reviewed; ADRs updated; SLO burn rate checked.

Appendix C — Review & Template Pack

These lightweight templates fit on a page and keep discussions grounded in user value and measurable outcomes.

C.1 SLO definition template

### Service: ___ (owner: ___)
**User Journey:** ___
**SLI(s):**
- Availability = success/total at ___
- Latency P95 < ___ ms measured at ___
- Freshness P99 < ___ s for ___
**SLO(s):** target ___ over ___ days
**Burn‑rate alerts:** fast ___×/5m; slow ___×/1h
**Dashboards:** links ___
**Runbook:** link ___

C.2 Blameless postmortem template

### Summary
**Impact:** ___ users; duration ___; business effect ___
**Timeline:** key events with timestamps
**Detection:** when/how; gaps
**Contributing factors:** technical + org
**What worked:** ___
**Actions:** owner, due date, test to verify
**Follow‑up review date:** ___

C.3 Architecture Decision Record (ADR) snippet

# ADR-___: Decision Title
**Context:** problem, constraints
**Decision:** chosen option
**Alternatives considered:** A, B (why rejected)
**Consequences:** pros/cons, risk
**Revisit when:** signal or metric threshold
