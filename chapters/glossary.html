Glossary

Glossary

A quick, alphabetized reference for the terms used throughout this course. Each entry includes a Plain definition, a Formal definition, and a short Example. Cross‑links let you jump to related ideas.

Tip: Use your browser’s find (Ctrl/Cmd‑F). Terms with related concepts link to each other (e.g., CRDT ↔ Event sourcing, Saga ↔ Compensating transaction).

ABCDEFGHIJKLMNOPQRSTUVWXYZ

A

Active‑active

Plain. Run the app in multiple regions at once, serving users from the nearest healthy one.

Formal. A multi‑region topology with concurrent primaries; reads are local; writes are coordinated via partitioning or global consensus; requires conflict handling or strict coordination.

Example. A global read API serves from the closest region while writes go to the partition owner; if a region fails, others continue without failover.

Active‑passive

Plain. One region is primary; another is on standby for failover.

Formal. Primary receives traffic; secondary replicates (usually async). Recovery point objective (RPO) > 0; recovery time objective (RTO) depends on failover automation.

Example. A database replicates to another region; you flip DNS only during incidents.

API gateway

Plain. The front door for APIs: routing, auth, rate limits.

Formal. A reverse‑proxy layer that centralizes cross‑cutting concerns (authN/Z, throttling, request shaping) and routes to backend services.

Example. Requests to /v1/orders are validated and routed to Orders; spikes are limited by a token bucket.

Autoscaling

Plain. Add/remove capacity automatically based on load.

Formal. Policy‑driven changes to replica count or instance size in response to metrics (CPU, QPS, queue depth), bounded by min/max limits.

Example. HPA increases pods when queue depth exceeds 500 for 5 minutes.

B

Backpressure

Plain. Slowing senders so the system doesn’t overload.

Formal. Flow‑control signals (queue limits, windowing, 429s) that bound concurrency or rate to match downstream capacity.

Example. Gateway returns 429 with Retry‑After when the order queue is full.

Bounded staleness

Plain. Data may be old, but only by a little.

Formal. A consistency model guaranteeing reads within a time/version bound of the latest write.

Example. Catalog pages guarantee prices are no older than 60 seconds; checkout revalidates.

Bulkhead

Plain. Isolate resources so one noisy neighbor can’t sink the ship.

Formal. Per‑route/tenant pools or queues to limit blast radius of contention and failures.

Example. Separate connection pools for /search and /checkout avoid starvation.

C

CAP theorem

Plain. During a network split you can’t have both perfect consistency and perfect availability.

Formal. In the presence of partitions, systems choose between linearizable operations (C) and total availability (A); outside partitions, both may be achievable.

Example. A checkout stays CP (reject writes on minority); a timeline feed remains AP and reconciles later.

Change data capture (CDC)

Plain. Stream DB changes to other systems.

Formal. Log‑based or trigger‑based extraction of inserts/updates/deletes to publish events or update read models.

Example. Order updates flow via CDC to a materialized view powering the account page.

Canary release

Plain. Roll out to a small slice, watch metrics, then expand.

Formal. Progressive exposure strategy gated by SLI deltas with automatic rollback on breach.

Example. Start at 1% mobile users; promote if P95 latency delta ≤ +10% and error delta ≤ +0.3%.

Cache‑aside

Plain. Read from cache; on miss, read DB and fill cache.

Formal. Application‑managed cache population with TTLs/invalidation; write path may bypass cache or update it.

Example. Product detail reads cache first; misses coalesce to avoid stampedes.

Cache stampede

Plain. Many clients hammer the origin when a hot key expires.

Formal. Thundering herd on TTL boundaries; mitigated by request coalescing, jittered TTLs, and background refresh.

Example. Add a single‑flight lock so only one request refreshes a hot page.

Circuit breaker

Plain. Fail fast when a dependency is sick.

Formal. A state machine (closed/open/half‑open) that blocks calls after error/latency thresholds and probes for recovery.

Example. Open after 2% errors for 3 windows; half‑open with 1 probe/s; close after 5 successes.

CRDT

Plain. A data type that converges across replicas without coordination.

Formal. Merge is associative, commutative, idempotent; replicas processing the same updates in any order reach the same state.

Example. Likes counter as a G‑Counter; removes cannot decrease value.

CQRS

Plain. Separate write and read models/paths.

Formal. Commands enforce invariants on a write model; queries read projections optimized for access patterns; eventual consistency tolerated on reads.

Example. Orders write to OLTP; a denormalized read model serves account pages.

D

Dead‑letter queue (DLQ)

Plain. A holding queue for messages your consumers couldn’t process.

Formal. Messages exceeding retry/age limits are moved to a DLQ for inspection and replay with fixes.

Example. Poison messages land in DLQ with error metadata; a job replays them after a bugfix.

Denormalization

Plain. Copy data into a shape that’s faster to read.

Formal. Store redundant fields or precomputed aggregates to reduce joins at query time; increases write amplification and staleness risk.

Example. Store seller name on product rows to avoid joins; update via CDC.

E

Error budget

Plain. How much failure you’re willing to tolerate in a period.

Formal. 1 − SLO (e.g., 0.1% of requests may fail per 30 days) used to pace risk and releases.

Example. If burn rate spikes, pause canary promotions and focus on stability.

Event sourcing

Plain. Store changes as an immutable log of events.

Formal. The event stream is the system of record; read models are deterministic projections; events are versioned and append‑only.

Example. Ledger stores credit/debit events and rebuilds balances.

ETag / If‑Match

Plain. A version tag that lets clients avoid lost updates.

Formal. Conditional requests (If‑Match/If‑None‑Match) that perform updates only if the server’s version matches the client’s.

Example. PUT /profile with If‑Match: "v7" rejects if another writer changed it.

Eventual consistency

Plain. Replicas may disagree temporarily but converge over time.

Formal. No real‑time ordering guarantee; clients may see stale data; convergence guaranteed if updates stop.

Example. Profile edits appear instantly to the editor (read‑my‑writes) but others see them after replicas catch up.

F

Fencing token

Plain. A monotonically increasing number that proves you’re the current lease holder.

Formal. Tokens issued by a coordinator; downstream stores reject writes with tokens lower than the last accepted, preventing split‑brain writes.

Example. A batch job includes token 42 in writes; if another takes over with token 43, old writes are ignored.

Feature flag

Plain. Toggle code paths at runtime for certain users.

Formal. A dynamic configuration controlling behavior by cohort; decouples deploy from release.

Example. Enable new checkout for 5% of traffic; roll back by flipping the flag.

G

gRPC

Plain. A high‑performance RPC protocol using HTTP/2 and Protobuf.

Formal. Contract‑first IDL with code‑generated clients/servers, bi‑directional streaming, and strict types.

Example. Low‑latency internal service calls with streaming updates.

H

HPA (Horizontal Pod Autoscaler)

Plain. Kubernetes controller that scales pods based on metrics.

Formal. Adjusts replica counts from minReplicas..maxReplicas using CPU/memory or custom metrics (QPS, queue depth).

Example. Scale Search pods from 3→12 when CPU > 70% for 5m.

I

Idempotency

Plain. Doing the same request twice has the same effect as doing it once.

Formal. A property of operations where repeated application yields no additional state change beyond the first.

Example. POST /orders with Idempotency‑Key: abc123 creates at most one order.

Ingress

Plain. Kubernetes HTTP(S) routing into the cluster.

Formal. An API object mapping host/path rules to Services, often backed by NGINX or a cloud load balancer.

Example. shop.example.com/checkout → CheckoutService.

J

Jitter

Plain. Add randomness to avoid synchronized spikes.

Formal. Randomized backoff/TTLs to de‑phase retries or expirations and reduce herd effects.

Example. Cache TTL = 300–360s instead of exactly 300s.

K

Kubernetes

Plain. A platform for running containers reliably at scale.

Formal. A control plane (API server, scheduler, controllers, etcd) that reconciles desired state (manifests) with actual cluster state via kubelets on nodes.

Example. A Deployment keeps 5 replicas of Orders alive and routes traffic via a Service.

L

Lamport clock

Plain. A way to order events without perfect time.

Formal. A logical counter that increases on local events and message receipts, establishing a happens‑before partial order.

Example. Use to break ties when merging edits without relying on wall‑clock.

LWW (Last‑Write‑Wins)

Plain. The newest write wins on conflict.

Formal. Conflict resolution by selecting the value with the maximum timestamp/vector clock.

Example. Profile picture URL uses LWW during offline merges.

M

Materialized view

Plain. A precomputed table for fast reads.

Formal. A persisted projection of base data, updated by triggers, jobs, or CDC; must handle rebuilds and staleness.

Example. order_view summarizes orders per customer; rebuild after schema change.

MVCC (Multi‑Version Concurrency Control)

Plain. Readers don’t block writers because the DB keeps snapshots.

Formal. Transactions read from a consistent snapshot; writes create new versions; isolation levels control anomalies.

Example. Long reads don’t stall short updates in Postgres.

N

NTP / chrony

Plain. Keep machine clocks in sync.

Formal. Network time protocols that discipline local clocks to reference servers, reducing skew that breaks time‑based logic.

Example. Prevents LWW errors from skewed clocks.

O

OpenTelemetry

Plain. A standard way to emit metrics, logs, and traces.

Formal. Vendor‑neutral SDKs, protocols, and semantic conventions for telemetry with context propagation.

Example. Add a trace id to every request and link spikes on dashboards to sample traces (exemplars).

P

Partitioning / sharding

Plain. Split data/work across nodes so it scales.

Formal. Horizontal division by key/range/hash to distribute load and storage; requires rebalancing and hot‑key mitigation.

Example. Orders sharded by customer_id with salted hot seller keys.

Postmortem (blameless)

Plain. A learning document after an incident, not a blame session.

Formal. A structured analysis of causes, detection, response, and actions with owners and due dates.

Example. Cert expiration incident leads to auto‑renewal and dashboards.

Q

Quorum (R + W > N)

Plain. Enough replicas must agree to read/write safely.

Formal. In N‑replica systems, choose read quorum R and write quorum W such that R + W > N to ensure overlap between reads and latest writes.

Example. With N=3, R=2 and W=2 ensures reads see a participant of the latest write.

R

Read‑my‑writes

Plain. After I write, my next reads show my change.

Formal. A session consistency guarantee binding a user’s subsequent reads to replicas that have applied their last write.

Example. Route the editor to the leader for 30 seconds after an update.

Read model

Plain. A storage shape designed for queries, not writes.

Formal. A projection optimized for read patterns, fed by CDC or events, often eventually consistent.

Example. An order_view table per customer for fast dashboards.

RPO / RTO

Plain. How much data you can afford to lose (RPO) and how fast you must recover (RTO).

Formal. Targets guiding backup/replication and failover design.

Example. RPO 1 minute; RTO 15 minutes for checkout.

S

Saga

Plain. A long transaction split into steps with undo actions.

Formal. A sequence of operations with compensations to restore invariants on failure; can be orchestrated or choreographed.

Example. Reserve stock → charge card → create shipment; on failure, refund and release.

Schema evolution

Plain. Changing data or event schemas without breaking clients.

Formal. Versioning rules (additive first), aliases, and deprecation windows; tested with consumer‑driven contracts.

Example. Add promo_code field; keep old readers working for 90 days.

Secrets manager

Plain. A vault for credentials with access controls and rotation.

Formal. KMS‑backed storage with audit and fine‑grained IAM; integrates with apps at runtime.

Example. Fetch DB creds via workload identity instead of storing in env files.

Service mesh

Plain. Sidecars handle networking concerns so apps can focus on business logic.

Formal. Data plane proxies + control plane for mTLS, retries, routing, and telemetry without app changes.

Example. Enforce mTLS and per‑route circuit breakers across services.

SLI

Plain. A measurement of user‑visible behavior.

Formal. A ratio or metric representing service performance (e.g., success/total, P95 latency).

Example. “95% of checkouts under 300 ms (client‑measured).”

SLO

Plain. The reliability target you promise to users.

Formal. A threshold for an SLI over a time window (e.g., ≥ 99.9% success over 30 days).

Example. Pause deploys if error budget burn exceeds 14× over 5 minutes.

Stampede (thundering herd)

Plain. Many clients do the same expensive thing at once.

Formal. Synchronization of retries/refreshes causing overload; mitigated by jitter, coalescing, and backoff.

Example. All caches expire at once; coalescing limits origin hits.

Strong consistency

Plain. After a write is confirmed, every read sees it.

Formal. Often linearizability: operations appear to occur atomically in real time.

Example. Money movement uses a single leader and quorum writes.

T

Tail‑based sampling

Plain. Keep the most interesting (slow/error) traces.

Formal. Decision to retain made after observing span attributes, not at ingress.

Example. Retain traces over 2× P95 or with 5xx status.

Throttling

Plain. Limit how fast clients can send work.

Formal. Token/leaky bucket algorithms that bound rate or concurrency; often return 429 with Retry‑After.

Example. Protect /login from brute‑force with per‑IP token buckets.

U

Unit economics

Plain. How much one user/request costs you.

Formal. Cost per unit of value (per request/message/user) guiding pricing and architecture choices.

Example. Trace and log budgets tied to cost per 1,000 requests.

V

Vector clock

Plain. Track causality across multiple writers.

Formal. A per‑replica counter vector; element‑wise maxima compute concurrency and detect conflicts.

Example. Reconciliation chooses winner by vector dominance; equal vectors mean concurrent updates.

W

Watermark

Plain. A streaming system’s best guess of how far it has progressed in event time.

Formal. A monotonically advancing marker that says “all events older than time T are likely processed,” enabling window triggers and late‑data handling.

Example. Fire window results at watermark = window_end − allowed_lateness.

Workload identity

Plain. Let apps prove who they are without static secrets.

Formal. Short‑lived credentials issued to runtime identities (pods/instances) via OIDC or metadata services.

Example. Pod exchanges its identity for a DB token; nothing is stored in code.
