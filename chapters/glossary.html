Glossary

A quick-reference to the vocabulary used throughout this book. Each entry includes a plain definition, a formal definition, and a short example. Cross-links point to related terms and chapters where you can go deeper.

A

Acceptance Criteria

Plain: The checklist that proves a feature is “done.”

Formal: Verifiable statements that map user requirements to testable outcomes (functional and non-functional), often tied to SLOs and SLIs.

Example: “P95 latency ≤ 200 ms for POST /checkout at 2k RPS with <0.1% errors.”

ABAC (Attribute-Based Access Control)

Plain: Authorize by attributes (department=finance), not just roles.

Formal: A model where policies evaluate subject, resource, action, and environment attributes, often alongside RBAC in a central policy engine.

Example: “Users with tenant_id=X and region=EU can read orders tagged tenant_id=X.”

ACID

Plain: The “safe transaction” rules for databases.

Formal: Atomicity, Consistency, Isolation, Durability; guarantees for single-node transactional systems, often traded against availability in distributed contexts (see CAP).

Example: A bank transfer either debits and credits or rolls back entirely.

API Gateway

Plain: The front door for your APIs.

Formal: A layer providing TLS termination, routing, authentication/authorization, rate limiting, and observability for north–south traffic; often paired with a WAF and service mesh.

Example: Public requests hit the gateway, which verifies a JWT and forwards to orders service.

Autoscaling

Plain: Automatically add or remove instances as load changes.

Formal: Control loops (HPA/ASG) that adjust replica counts based on signals like SLIs (latency, inflight, queue lag) with stabilization windows and safety caps.

Example: Scale workers up when consumer lag grows; scale down after 10 minutes of low lag.

Availability

Plain: The fraction of time your system works.

Formal: Probability a request completes successfully within the SLO bounds over a period; expressed as “nines.”

Example: 99.95% monthly availability ≈ 21.6 minutes of allowed unavailability.

B

Backpressure

Plain: Telling callers to slow down when you’re overloaded.

Formal: Flow-control mechanisms that bound queue growth and response time, often via limited concurrency, 429s, or server-side shedding; complements retries and circuit breakers.

Example: When queue depth exceeds 100, the service returns 429 with a Retry-After header.

Bulkhead

Plain: Partitioning capacity so one failure doesn’t sink the ship.

Formal: Resource isolation (threads, connections, queues) per dependency or priority class to contain blast radius.

Example: Separate worker pools for “checkout” vs “reports” so analytics load can’t starve orders.

BFF (Backend for Frontend)

Plain: A server that speaks for the web/mobile client.

Formal: An edge service that holds tokens, composes data, and exposes client-optimized endpoints to minimize client complexity and exposure of secrets (see OIDC).

Example: SPA calls BFF; BFF exchanges code for tokens and calls internal APIs.

C

CAP Theorem

Plain: In a partition, you choose which promise to keep.

Formal: In a distributed system with network partitions, you can provide either availability (every request gets a response) or linearizable consistency, but not both simultaneously.

Example: A geo-replicated KV may prefer availability with eventual consistency during partitions.

Cache-Control

Plain: HTTP headers that tell browsers/CDNs how to cache.

Formal: Directives like max-age, s-maxage, stale-while-revalidate, and private/no-store governing freshness and privacy at shared caches.

Example: Cache-Control: public, max-age=60, s-maxage=300, stale-while-revalidate=30.

CDC (Change Data Capture)

Plain: Stream database changes to other systems.

Formal: Log-based or trigger-based extraction of row-level changes to build replicas, caches, or event streams; often paired with the outbox pattern.

Example: ProductUpdated events feed the search index and invalidate caches.

Circuit Breaker

Plain: A switch that opens when calls fail too much.

Formal: A protective pattern that transitions between closed, open, and half-open states to prevent cascading failures; complements backpressure and retries.

Example: After 20% failures to payments service, stop calling for 30 seconds and degrade to “save cart.”

Consistent Hashing

Plain: A way to shard so few keys move when nodes change.

Formal: Hash-space ring mapping keys to nodes via virtual replicas to minimize redistribution on membership changes.

Example: Redis cluster with 16k hash slots and virtual nodes.

Consumer Lag

Plain: How far your consumers are behind the stream.

Formal: Difference between the latest produced offset and the committed offset; a scaling and health SLI for streaming systems.

Example: Autoscale consumers when lag > 10k messages for 5 minutes.

D

DLQ (Dead Letter Queue)

Plain: A parking lot for failed messages.

Formal: A queue where messages that exceed retry limits are routed for quarantine, inspection, and redrive.

Example: Malformed webhooks move to DLQ with cause; an operator fixes and replays.

Domain / Bounded Context

Plain: A slice of the business where words have specific meanings.

Formal: DDD concept: a boundary within which a model applies consistently; contracts between contexts are explicit APIs or events.

Example: Orders and Catalog are separate bounded contexts with their own databases.

E

Envelope Encryption

Plain: Encrypt data with a data key that’s itself encrypted by a master key.

Formal: Hierarchical key management where DEKs encrypt payloads and are wrapped by KMS/HSM-protected CMKs; supports rotation and limited blast radius.

Example: Each tenant has its own DEK for PII columns; CMKs rotate quarterly.

Eventual Consistency

Plain: Data may be briefly stale but will converge.

Formal: A consistency model in which replicas asynchronously propagate updates; reads are not guaranteed to see the latest write immediately but will do so given sufficient time.

Example: Search results reflect catalog updates within 5 seconds (freshness SLA).

F

Feature Flag

Plain: A runtime switch for behavior.

Formal: Conditional evaluation that targets users/tenants to enable variants, enabling canary and progressive delivery.

Example: Enable new search for 5% of EU tenants; auto-rollback on SLI degradation.

Freshness SLA

Plain: How stale a view is allowed to be.

Formal: A bound on the age of data in derived systems (caches, indexes) measured by lag or timestamps.

Example: “Catalog changes visible in search within 5 seconds, 99% of the time.”

G

GDPR

Plain: EU data protection rules.

Formal: Regulation requiring lawful basis, data minimization, subject rights, breach notification, and controls for international transfers.

Example: Implement erasure for user data and record data processing purposes.

H

Hedged Requests

Plain: Send a backup request when one is slow.

Formal: Duplicate a call after a delay (e.g., P95 threshold) and take the first successful response; bounded by budgets to avoid overload.

Example: Duplicate reads at 100 ms if no response yet; cancel the loser.

HPA (Horizontal Pod Autoscaler)

Plain: Kubernetes’ autoscaling for pods.

Formal: Adjusts replica counts based on metrics (CPU, custom) with stabilization and min/max bounds; see Autoscaling.

Example: Scale API pods from 8→24 on P95 latency > 180 ms.

I

Idempotency

Plain: Safe to repeat the same request.

Formal: An operation that produces the same effect when applied multiple times; often enforced by idempotency keys, upserts, or unique constraints.

Example: POST /orders requires Idempotency-Key so retries don’t duplicate charges.

Inflight Requests

Plain: How many requests are being processed right now.

Formal: Concurrency metric correlated with queueing; useful autoscaling signal with backpressure.

Example: Target 2 inflight per core to keep latency stable.

J

JWT (JSON Web Token)

Plain: A signed token your APIs can verify locally.

Formal: JWS-signed claims (e.g., RS256/ES256) including iss, aud, and exp; validated against issuer JWKS (see OIDC).

Example: API checks signature, audience, and expiry before authorizing orders:read.

K

KMS (Key Management Service)

Plain: Managed keys for encryption and signing.

Formal: A control-plane for generating, storing, and auditing cryptographic keys; integrates with envelope encryption.

Example: Only the profile service can use the CMK to decrypt PII; separate admins manage key rotation.

L

Latency Percentiles (P50/P95/P99)

Plain: How fast requests are for most users, and for the slow tail.

Formal: Quantiles of the latency distribution measured via histograms; tails drive user experience and capacity planning.

Example: Optimize P95 before P50; watch P99 during canaries.

M

mTLS (Mutual TLS)

Plain: Both client and server prove who they are.

Formal: TLS with client certificates for workload identity; foundational for zero trust and service meshes.

Example: Orders and Payments services authenticate with SPIFFE-based mTLS.

Monotonic IDs

Plain: Increasing IDs that can cause hot partitions if used for sharding.

Formal: Sequential identifiers that concentrate writes on recent key ranges; mitigate with hashing or time-bucketed shards.

Example: Use hash(order_id) mod N for shard placement, not raw ID.

N

Nines

Plain: Shorthand for availability levels.

Formal: 99% (two nines) ≈ 3.65 days/yr downtime; 99.99% (four nines) ≈ 52.6 minutes/yr.

Example: Choose nines based on user tolerance and cost of redundancy.

O

Observability

Plain: Seeing what the system is doing and why.

Formal: The ability to infer internal state from outputs—metrics, logs, and traces—often via OpenTelemetry; anchored on user-centric SLIs.

Example: Dashboards show checkout success, P95 latency, and link exemplars to traces.

OIDC (OpenID Connect)

Plain: Identity layer on top of OAuth 2.0.

Formal: Adds an ID Token and user info endpoints to OAuth2 flows; recommended with Authorization Code + PKCE for web/mobile.

Example: SPA uses BFF to exchange code for tokens; APIs validate access tokens.

Outbox Pattern

Plain: Save your event with your data so they can’t get out of sync.

Formal: Write domain data and an event in the same transaction; a relay publishes the event to a log/queue, ensuring at-least-once delivery to downstreams like caches or search.

Example: On order update, write order row and order_updated outbox record; relay emits Kafka event.

P

Knee of the Curve

Plain: The load point where latency starts rising sharply.

Formal: Transition from stable to queuing-dominated region in throughput–latency plots; operating target sits 20–40% below.

Example: Capacity plan sets 8k RPS ceiling when knee occurs at ~10k RPS.

Progressive Delivery

Plain: Roll out changes gradually with metrics watching.

Formal: Techniques (canary, blue/green, flags) to increment exposure based on SLIs, enabling fast rollback.

Example: 1% → 10% → 25% → 100% rollout with automatic abort on error budget burn.

Q

Quorum

Plain: “Enough” nodes to agree.

Formal: Majority or configured subset of replicas required to accept reads/writes, enabling consistency bounds in replicated systems.

Example: Write requires W=2 of 3; read requires R=2 of 3; R+W>N ensures no stale reads.

R

RBAC (Role-Based Access Control)

Plain: Permissions granted by role names.

Formal: Users assume roles mapped to permissions; often combined with ABAC for tenant/region scoping.

Example: Role orders.admin can refund orders; orders.viewer cannot.

Retries

Plain: Try again when a request fails.

Formal: Bounded, exponential backoff with jitter; only for idempotent operations; coordinated with backpressure to avoid amplification.

Example: 3 retries at 100/200/400 ms with jitter for 5xx and timeouts.

RPO / RTO

Plain: How much data you can lose (RPO) and how long you can be down (RTO).

Formal: Recovery Point Objective (max acceptable data loss) and Recovery Time Objective (max acceptable downtime) for disaster recovery.

Example: RPO 5 minutes, RTO 30 minutes with multi-AZ replicas and scripted failover.

S

SLA (Service Level Agreement)

Plain: The public promise (often contractual) of service quality.

Formal: An external commitment including remedies; informed by internal SLOs and measured by SLIs.

Example: “99.9% monthly availability” with service credits for breaches.

SLI (Service Level Indicator)

Plain: A metric users care about.

Formal: A carefully defined measurement of service performance (e.g., success ratio, P95 latency) used to judge SLO attainment.

Example: “Checkout success = 2xx/total over 5 minutes, tenant-scoped.”

SLO (Service Level Objective)

Plain: Your internal target for an SLI.

Formal: A quantitative goal for an SLI over a time window; defines error budgets that drive risk decisions.

Example: “P95 latency ≤ 200 ms for GET /feed weekly.”

Service Mesh

Plain: Infrastructure that adds mTLS, retries, and policy between services.

Formal: Data-plane proxies with a control plane to enforce identity, telemetry, and traffic policies for east–west calls.

Example: Sidecars inject mTLS and limit egress destinations per service.

stale-while-revalidate

Plain: Serve slightly old content while fetching fresh in the background.

Formal: An HTTP caching extension allowing shared caches to return stale responses for a window while revalidating with ETag or timestamps.

Example: CDN serves cached product page and refreshes it asynchronously.

Strangler Pattern

Plain: Replace a legacy system piece by piece.

Formal: Route flows through a seam (gateway); new services take over capabilities gradually; old code is deleted when unused.

Example: Move catalog reads/writes out of the monolith first, then orders.

T

Tail Latency

Plain: The slowest requests users feel.

Formal: High-percentile latency (P95/P99) dominating perceived performance; amplified by fan-out and queueing.

Example: Use hedged requests and caching to tame tails.

TTL (Time To Live)

Plain: How long a cached item stays fresh.

Formal: Expiry horizon for cache entries; combined with versioned keys or event-driven invalidation.

Example: product:{id}:v3 cached 5 minutes with TTL jitter to avoid stampedes.

Tolerant Reader

Plain: A client that ignores fields it doesn’t know yet.

Formal: Consumer design that survives additive schema changes by defaulting unknowns and ignoring extras; key to compatibility.

Example: JSON client ignores new summary property until it’s ready to use it.

U

Unit Cost

Plain: Cost per unit of user value.

Formal: Normalized spend (e.g., ₹/1k requests, ₹/order) used to evaluate scaling, caching, and architecture trade-offs.

Example: CDN offload halves ₹/1k product views while meeting SLOs.

Upsert

Plain: Insert or update in one step.

Formal: A write that creates a row if absent or updates it atomically if present; common building block for idempotency.

Example: INSERT ... ON CONFLICT DO UPDATE for idempotent counters.

V

Versioning (APIs & Schemas)

Plain: A plan for change without breakage.

Formal: SemVer for libraries; /vN/ or media-type versions for HTTP; reserved tags and additive changes for Protobuf/Avro; expand/contract for DB.

Example: Keep v1 and v2 APIs live for 90 days; publish deprecation dates.

W

WAF (Web Application Firewall)

Plain: A filter that blocks common web attacks.

Formal: L7 ruleset (managed or custom) to mitigate OWASP Top 10 patterns; sits with API gateways.

Example: Block SQLi probes and rate-limit suspicious IPs at the edge.

Z

Zero Trust

Plain: Trust no network, verify each request.

Formal: Architecture that authenticates users/devices/workloads and enforces least-privilege access at application edges and service boundaries, often using mTLS and context-aware policies.

Example: Internal services require identity via mesh-issued certificates; user access via identity-aware proxies.
